  キテクチャCPUや、bitやwordレベルでPCを追跡するCPU向けに調整されています。

  関数やマクロで名前が変更になったものがあります。使用するドライバについては全て

      cpu_setbankhandler_*  -> memory_set_bankhandler_*

- ROMロードインターフェースを大幅に書き換えました。
  以前のROM読みこみシステムには限界があって、バイトの順序付け関連でソースが混乱
  していました。新システムはこれに対処するのが目的です。全ドライバは新システムを
  利用するように完全にコンバートされています。旧マクロは廃止します。

  ROM_REGIONマクロに、領域用の各種フラグを保持する3番目のパラメータが追加されて
  います。このフラグは領域のデータ幅とバイト並び順(endianness)を指定できるように
  拡張されています（関連CPUからあらかじめそれらを獲得するCPU領域は除きます）。
  ここでの情報は、ターゲットとなるプラットフォーム独自のバイトの並び順に対して、
  正しくデータのバイトスワップを行うため、領域のロードが終わってから使用します。

  追加の領域フラグROMREGION_INVERTが利用できます：これは領域中の全てのデータを
  ビット反転(NOT演算)します。またROMREGION_ERASEはROM読み込み前に特定のバイト値
  で領域を初期化するのに利用します。

  ROMの読み込みについては、以前のROM_LOAD_XXX_EVEN/ODD系マクロはもう使いません。
  領域のそれぞれのバイトにROMをロードするには、新しいROM_LOAD16_BYTEマクロを使っ
  て下さい。全てのデータは自然なアライメントと領域の並び順でロードされます。
  ROMのロード後のみ、データはCPUが必要な形式に変換されます。

  CPU用のデータを非CPU領域にロードする場合(例えば、68000コードをREGION_USER1など
  にロードするなど)、その領域をCPUと同じものであると示す必要があります。68000コー
  ドを含むユーザ領域については、その領域の宣言にROM_REGION16_BE()マクロを使って
  下さい。ただし、ROMローダーは自動的にCPUインターフェースで必要な情報を割り出す
  ので、REGION_CPUxxを使う必要は必ずしもありません。

  古いドライバをコンバートする際には、これらの変更で99%のものに対応できます。
  (オフセットフィールドを示す(0)また(1)は、偶数の(0)または奇数の(1)のアドレスと

      ROM_LOAD_EVEN         -> ROM_LOAD16_BYTE(0)
      ROM_LOAD_ODD          -> ROM_LOAD16_BYTE(1)
      ROM_LOAD_WIDE         -> ROM_LOAD16_WORD
      ROM_LOAD_WIDE_SWAP    -> ROM_LOAD16_WORD_SWAP

      ROM_LOAD_GFX_EVEN     -> ROM_LOAD16_BYTE(0)
      ROM_LOAD_GFX_ODD      -> ROM_LOAD16_BYTE(1)
      ROM_LOAD_GFX_SWAP     -> ROM_LOAD16_WORD_SWAP

      ROM_LOAD_V20_EVEN     -> ROM_LOAD16_BYTE(1)
      ROM_LOAD_V20_ODD      -> ROM_LOAD16_BYTE(0)
      ROM_LOAD_V20_WIDE     -> ROM_LOAD16_WORD

  これらの新マクロは、さらに柔軟性が向上したROMロード命令系で構成されています。
  ROMデータは1〜16バイトの"グループ"としてロードされ、それぞれ使用していない部分
  には0〜15バイトのスペースが入ります。このデータは通常の並び順でロードすること
  もできますし、逆順で読み込むことも可能です。また、ロードされたそれぞれのバイト
  部分は既にロード済みのデータと区別できるようにマスク、シフトができます。

  さらに、ほとんどのROM_RELOADとROM_CONTINUEは、ROMをロードする際に完全に同じ引数
  を取っていたため、これらのマクロでは以前の命令での動作を継承するようにしました。

  きれいさっぱり片付けました。例えば、データをロードするときにROM_LOAD16_BYTEを使い、
　続けてROM_CONTINUEを使って、個別のオフセットで追加のデータを読み込めば、ROMロー
  ダーが個々(訳注：一つおき?)のデータ読み込みを続けてくれます。

  最後に、未使用、または複製されたROMデータを埋めるのに便利なエントリーを2つ追加
  しました：

      ROM_FILL( offset, length, byteval)
      ROM_COPY( source_region, source_offset, offset, length )

  ROM_FILLは単純に、特定のバイト値に領域ブロックをクリアします。
  ROM_COPYは同一か、他の領域からあらかじめロードされているデータをコピーします。
  [Aaron Giles]

-- 始めにスポンサーからの言葉 --

しばらく前に私（ニコラ）は「Lotto Fun」をギャンブルゲームと勘違いして削除してし
まいました。その後よくよく調べてみると、これは普通の技を競うゲームだということが
わかりました。そのためこれを復活させます。これは私のミスで本当にすいません。

ついでに言うと、ギャンブルゲームをはずすかどうかは、モラルや法的な問題ではなく
単に技術的なものです。ギャンブルマシンはビデオゲームではなく、MAMEが機械式のピ
ンボールをサポートしないように(ただしソフトウェアのピンボールはサポートします)、
ギャンブル機はサポートしません(ただしギャンブルをテーマにしたものはサポートし
ます)。機械式のピンボールとギャンブル機はビデオゲームのある所では大抵同じ場所に
置かれていますが、このことでこれらのゲームをひとくくりにするわけではありません。
ギャンブル機とビデオゲームの線引きは何となく行っているのではないのです：
大抵のギャンブル機はプレイするのに普通のゲームよりも金がかかります(例えば10コイ
ンスタートなど)。また、ギャンブル機には払い出し率の設定があり、勝敗はプレイヤー
のテクニックによるのではなく、完全にマシン次第なのです。

-- それではいつものプログラミングにもどりましょう --

  - struct RunningMachineにvisible_areaを追加。ドライバではMachine->drv->visiblearea

    れるので、画面を回転させる必要のあるゲームでは、これ以上の処理は必要なくなります。
    OS依存コードでは、もし必要な場合はビットマップのより小さい部分を表示することも可
    能ですが、その逆にビットマップよりも大きい部分を指定するとゴミ部分が表示されてし
    まうので注意してください。
    この関数はまず、「set_ui_visarea()」を呼び出してコアに対して、ビットマップの実際
    にどの部分が表示されるのかを設定する必要があります(これは恐らく必要なものよりも
    小さくなるでしょう)。このため、ユーザーインターフェースはそれに応じて描画すること
    が出来ます。

    scrbitmapを描画するのをハード組み込みにしてはいけません。
  - struct GameOptionsにvector_widthとvector_heightフィールドを追加。
    OS依存コードはベクターゲームで別な解像度を設定するために、これらを初期化する
    必要があります。もし0にされた場合はデフォルトの640x480が使われます。
  -「osd_create_display()」も画面にビットマップを生成せず、コアによって処理されます。

- drawgfx()の新モード追加：TRANSPARENCY_NONE_RAW、TRANSPARENCY_PEN_RAW、
  TRANSPARENCY_PENS_RAW、TRANSPARENCY_THROUGH_RAW。
  この目的は、「未処理(raw)」のカラースペース(再マップされていないペン)での操作を
  可能にして、パレットが終端かどうかを調べるのに機能します。これは当たり判定を行っ
  たり、前景と背景のブレンドをする多くのゲームで便利な機能です(例：パレットインデッ
  クスが((fgpen << 4) | bgpen)の時など)。
  このモードを使うときは、「color」パラメータはそれ以上パレットを参照しなくなりま
  す。その代わり、パラメータには現在ドロー中のグラフィックでの未処理ペンの値が追加
  されます。このように、もし背景レイヤーで0〜63番のペンを使っているときや、前景レイ
  ヤーで、64〜127番のペンを使っている場合に、背景レイヤードロー時の0番の色、前景レ
  イヤードロー時の64番の色を指定できます。
  また、「copybitmap_remap()」と「copyscrollbitmap_remap()」も追加しました。
  これらもオリジナルと同じ引数をとりますが、TRANSPARENCY_NONEがTRANSPARENCY_NONE_RAW
  を意味するというような仮定をしません。
  そのため、最後の最後までrawパレットのインデックスで全ての処理が出来ます。さらに、
  「copybitmap_remap(TRANSPARENCY_NONE)」で結果のコピーと、終端での再マップステップ
  を行います。
  最後に「copybitmap()」内のみでサポートされる新規追加モード、TRANSPARENCY_BLEND
  とTRANSPARENCY_BLEND_RAWについて。この2つのモードでは、ソースのビットマップが
  「color」パラメータにより左へシフトされ、目的のビットマップへORブレンドされます。
  この機能により、前景と背景のペンインデックスの重ね合わせによってパレットを参照する
  ゲームで、パレットインデックスの算出や参照が可能です。[Aaron Giles]

　渡される関数は(col,row)->tile_indexのマッピングを行います。2つのスタンダード関数は
　一般的な用途で提供されます。現在不調のnamcos2.c以外のドライバは全て、新しいAPIを
　使用するように変換されました。[Phil Stroffolino]

　デートしました。古い「function(int offset,int data)」のスタイルはもう使いません。
